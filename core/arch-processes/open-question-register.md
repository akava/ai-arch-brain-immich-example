# Open Questions Register

The live registry of what is not yet known — `OQ#n` / `Q#n`, cited across the brain (`core/0.ARCH-METAMODEL.md` → Shared registries). Open questions only: a concluded question's row is **removed** — its answer lands where it belongs (ARCH-TARGET, an ADR, or via the register action that closed it). The brain is the record; no answered-questions archive is kept here.

1. **Multi-node HA posture — shared-store topology now documented, HA/failover still absent.** **Partially answered by D3:** the first-party scaling guide now confirms the multi-node *topology* — parallel `immich-server` instances must share **one Postgres, one Redis, and identical file mounts**, the API worker may be disabled on background-only instances, and Kubernetes scaling is "incrementing replicas" (BASELINE §3.4; `input/systems/immich/immich-scaling-guide.md:9,13,15,19,23`). **Still open:** the guide explicitly **gives no capacity/sizing instructions** ("left as an exercise for the reader"), and there is **still no HA, replication, or failover path** for the three shared stores — the availability risk (R-01) is unresolved, and the shared stores are now confirmed as a **single point of failure** for every instance behind them. Also unaddressed: replica coordination and rolling-upgrade behaviour against the shared Postgres (R-05). Material to the 99.9% availability target (`core/1.ARCH-CONTEXT.md` → Goals, client-stated fiction). (Opened 2026-07-11 D1, source: `2026-07-11-immich-architecture-overview-synthesis.md` T5, C1; partial answer 2026-07-11 D3, source: `2026-07-11-immich-scale-up-design-synthesis.md` T14/T17. HA design + sizing need external evidence/operator design beyond this doc set.)

2. **ML-inference throughput remains unquantified** (hardware floors + coarse concurrency heuristics now documented). `immich-machine-learning` is documented as memory-intensive and separately deployable. **Partially answered by D2:** hardware *floors* for GPU acceleration are on record — CUDA compute capability 5.2+ and driver ≥ 545, ROCm ≥ 35 GiB disk, RKNN 2–3 threads for RK3576/RK3588, multi-GPU via `MACHINE_LEARNING_DEVICE_IDS`/`MACHINE_LEARNING_WORKERS` (BASELINE §3.4). **Further partially answered by D3:** the FAQ adds coarse concurrency heuristics — **2–3 concurrent jobs max a CPU, ≤16 with a GPU**, transcoding threads 1–2, lighter `buffalo_s` face model — and the tuning defaults `MACHINE_LEARNING_WORKERS=1`, `MODEL_TTL=300s`, request-threads = CPU cores (BASELINE §3.1/§3.4; `input/systems/immich/immich-faq-scaling-excerpts.md:49,67`, `immich-environment-variables.md:83-89`). **Still open:** per-node/per-GPU **throughput in assets/sec** and memory footprint per model — no figures exist to size the 1M-assets-in-72h onboarding target. Material to the D3 ML-throughput target (client-stated fiction) and R-03. (Opened 2026-07-11 D1, source: `2026-07-11-immich-architecture-overview-synthesis.md` T4; partial answers 2026-07-11 D2 `2026-07-11-immich-ml-media-deep-dive-synthesis.md`, 2026-07-11 D3 `2026-07-11-immich-scale-up-design-synthesis.md` T17. Resolution needs telemetry/benchmarking beyond the first-party doc set.)

3. **Remote-ML channel has no built-in security or transport protection.** When `immich-machine-learning` runs on a separate host, the server sends **image previews** (user photo content) to it over port 3003, and the docs state the ML container "lacks built-in security measures," warning against public/paid-cloud placement without added protection (`input/systems/immich/immich-remote-machine-learning.md`; BASELINE §1.2). In a managed-hosting topology where the ML tier is a separate scaling unit, this leaves the operator to supply auth, transport encryption, and network isolation — none of which the docs specify. Material to the managed-hosting security posture; needs an operator-side control design (now tracked as risk **R-06** and requirement **NFR-006**; hardening direction set by ADR-004, proposed). (Opened 2026-07-11 D2; source: `core/artifacts/logs/2026-07-11-immich-ml-media-deep-dive-synthesis.md`.)

4. **No documented API-key surface; session-management documented only indirectly.** The fetched Immich system-settings page contains **no dedicated API-key or session-management section** (the fetch report flagged it, `input/systems/immich/immich-system-settings-auth.md:35`). D4 brief Q3 asks for password/session/API-key documentation: **password auth** and one **partial session behavior** are documented and captured in BASELINE §1.6 (password login is disableable instance-wide incl. admin; auth-setting changes do **not** invalidate existing sessions), but **API keys** (the standard non-interactive/service-account credential path) and **per-session invalidation / forced logout** (the standard incident-response control) are absent from the fetched docs. Material to managed hosting — service-account automation and the ability to revoke a compromised session are both operator concerns with no documented Immich mechanism here. What exists is in the baseline; this is the gap. (Opened 2026-07-12 D4; source: `core/artifacts/logs/2026-07-12-immich-identity-access-synthesis.md` T23. Resolution needs the full API-key/session docs or source inspection beyond the fetched settings page.)
