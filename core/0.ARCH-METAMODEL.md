# 0.ARCH-METAMODEL.md — Architecture Description Model

The constitution of this brain's architecture description. Structure changes — sections, viewpoints, status tags, ID schemes, embedding conventions — land **here first**, then propagate: this file is the gate that keeps the description layers consistent. Such changes are machinery, not architecture: their record is their commit, never an ADR (`core/AGENT-RULES.md` → Change history).

## One system, three spaces

The **system-of-interest** is the system this engagement designs and evolves — named in the `core/1.ARCH-CONTEXT.md` engagement card (TODO: name it there when the engagement starts). It has one description, split across three spaces:

| Space | Holds | Files |
|-------|-------|-------|
| Problem | the engagement card + who needs what and why — stakeholders, goals, scope & constraints, essential requirements | `core/1.ARCH-CONTEXT.md` |
| Solution | the system at two times + the path between | `core/2.ARCH-BASELINE.md` (as-is) · `core/3.ARCH-TARGET.md` (target) · `core/transitions/` (one file per delivery phase) |
| Process | how the architecture work runs | `core/arch-processes/` — enters the solution description only as §5 constraints or evidence-confidence qualifiers |

Framework grounding: TOGAF-ADM shape at repo level (baseline → target via transitions; gap analysis = §-to-§ comparison); a tailored **arc42 profile** for the solution files (table below); **ArchiMate motivation** elements for the problem file; **the C4 model** (context/container zoom) inside §1.2 — always spelled out in full, because bare `C1…C5` are synthesis-conflict IDs in this repo; ISO/IEC/IEEE 42010 vocabulary throughout.

**SAFe correspondence** (delivery framing): Architectural Runway ≈ `core/2.ARCH-BASELINE.md` — the present-tense foundation already built, which features consume and enablers extend; Architectural Vision ≈ `core/3.ARCH-TARGET.md` — the ongoing technical direction spanning milestones; a transition = the vision slice one milestone delivers plus the enabler package that carries it out. SAFe's "PI" is not used here — the delivery timebox is a **milestone** (`Mn`), and the phase file is an **arch transition**.

## Content boundary rule (where facts land)

`core/` holds the architecture of the system under design — patterns, components, contracts, NFRs, constraints, decisions. How the architecture work itself runs — engagement posture, cadence, escalation discipline, reporting instruments, and the ownership map (architecture × org) — homes in `core/arch-processes/`, an evolving folder. Only genuinely exploratory material and the programme people roster stay in `lab/`. Within `core/`, facts split by time: as-is estate facts (including the component ownership/accountability map) home in `core/2.ARCH-BASELINE.md`, phase-scoped facts (scope, work streams, coexistence approach) in the active `core/transitions/` file, target rules in `core/3.ARCH-TARGET.md` — one home per fact, cross-referenced. BASELINE and TARGET share one section skeleton (same §N = same concern, tense decides the file) and **problem-space material — stakeholder needs, goals, requirement statements (`NFR-`/`FR-`/`CON-NNN`) — homes in `core/1.ARCH-CONTEXT.md`**, cited by ID from the solution files. Apply the boundary at intake (synthesis theme homing), not only at fitness review.

**Subject test — the architecture vs the work on it.** A fact about the *architecture* (its elements, behaviour, gaps, component ownership) lands in the architecture layers; a fact about *the work on the architecture* (access, capacity, review progress, cadence, reporting) lands in `core/arch-processes/`. Process limits never enter ARCH-BASELINE as facts — they appear there only as evidence-confidence qualifiers on architectural entries.

**Process vs deliverable.** Each architecture layer has a work stream in `core/arch-processes/work-streams/` — the **process** whose goal is to populate that layer: Architecture Review → `core/2.ARCH-BASELINE.md`; Target Architecture → `core/3.ARCH-TARGET.md`; Delivery Enablement → the active `core/transitions/` file. Stream files hold goals, work status, and blockers; every durable fact lands only in the layer the stream populates — the stream links, never restates.

**Time boundary test** — classify every fact before landing it:

- **Is / runs / observed today** → `core/2.ARCH-BASELINE.md`: as-is facts, as-built designs, observed metrics, current ownership, current gaps.
- **Must be / become — a rule work is validated against** → `core/3.ARCH-TARGET.md`: patterns, sanctioned tech, NFR targets, conventions, binding gates; status-tagged. May be architect-directional without an ADR — an ADR is required when direction firms into commitment, typically during a transition (→ Status convention).
- **Scoped to a named delivery phase** → the active `core/transitions/` file: phase scope, work streams, baseline→target deltas, coexistence/flag approach, sequencing.

One statement often carries facts for several layers — split it: "checkout takes 4–6s and must be ≤2s" is an ARCH-BASELINE observation plus an ARCH-TARGET §3 target plus, when phase-scoped, a transition delta. A to-be-replaced component stays described in ARCH-BASELINE, is absent from ARCH-TARGET's sanctioned set, and its replacement is the transition's delta. ARCH-BASELINE changes only when reality changes — a phase delivers or discovery corrects the picture — never because a target was decided. Two carve-outs: a governance rule that genuinely constrains the architecture (e.g. gates a deployment path) is promoted into `core/` (ARCH-TARGET §5), and a significant process/governance *decision* is still recorded as an ADR (`decision-record-agent`).

## Entry anatomy

`[status]` **fact** — inline ID refs (`NFR-`/`FR-`/`CON-`/`R-`/`ADR-`/`AI-NN`) + **one** compact provenance citation (`path §/T-row`). No corroboration chains, no supersession narratives, no edit commentary — git and ADR bodies own history. Dates appear only as evidence qualifiers.

## Diagram convention

Solution files may embed **Mermaid** diagrams (fenced ` ```mermaid ` blocks), C4-informed zoom per the framework grounding (§1.1 context, §1.2 container). Rules:

- **Illustrative only — text entries are authoritative.** A diagram renders existing populated entries and introduces no new elements, facts, or relationships; each carries a lead-in stating what it renders (with the status tag and ID citations of its sources).
- **Names per `core/GLOSSARY.md`**; edges and nodes carry their ID refs (`ADR-`/`NFR-`/`E-`/`OQ#`) like any entry.
- **Line semantics:** solid = data/default paths; dashed = control planes, optional configurations, and exceptions.
- **Topology or sequence only.** Registry-shaped content (stakeholders, requirements, tech stack, NFR targets) stays tabular; a diagram that would be a list with boxes around it is a table.
- **No diagram for unpopulated sections** — a TODO section gets no diagram (nothing to render without invention).
- The committed reference-architecture diagram in `input/`, when one exists, remains the authoritative diagram (ARCH-TARGET → Description rules); embedded Mermaid never overrides it.

## Status convention

- **`[Confirmed]`** — verified against an authoritative source in `input/`, or settled by an approved ADR. No ADR citation is required on the entry; its provenance is the source.
- **`[Tentative]`** — architect-directional content, or backed by a **proposed** ADR the team is **leaning to adopt** (cite as `(ADR-NNN, proposed)`). Written now so the description reflects the built/intended system; may change until approved (`core/AGENT-RULES.md` §8). **Target content may be directional without any ADR** — an ADR is required when direction firms into commitment, typically during a transition. Never withheld merely because an ADR is unapproved.
- **`[Pending-verification]`** — either (a) drawn from an external reference and not yet confirmed by an authoritative `input/` source; **or** (b) the recommended option of a proposed ADR where the decision is **genuinely still open** — a lower-commitment signal than `[Tentative]`, cited as `(ADR-NNN, proposed — recommended option, decision still open)`.
- **`[TBD]`** — not yet defined, not backed by any ADR or architect direction. Surface the gap; never fill it with assumptions.

## Source & naming rules

- **Only content inside `input/` (committed to this repository) is authoritative.** External documents are references; content drawn from them stays `[Pending-verification]` until an authoritative `input/` source confirms it.
- Use only IDs and names from the reference-architecture diagram: element names with their diagram number, layer numbers, topic names. Requirement IDs: `NFR-NNN` (non-functional), `FR-NNN` (functional), `CON-NNN` (constraint) — the legacy `REQ-…-NNN` form in ADR bodies and run logs resolves via the requirements registry's "formerly" column (`core/1.ARCH-CONTEXT.md`). Decisions: `ADR-NNN`.

## Integrity rules

- Structured outputs must not contradict populated entries of the solution files.
- A `[TBD]` entry means the gap must be surfaced, not filled with assumptions.
- Populated entries carry a status tag and a provenance citation (see Entry anatomy).
- One home per fact; cross-reference, never restate. The subject test, then the time test decide the home (→ Content boundary rule).
- § parity between BASELINE and TARGET is checked at fitness review; skeleton changes go through this file.
- Requirements split by altitude: the **high-level, intentional** (capability-level) statements are defined in the `core/1.ARCH-CONTEXT.md` registry; **milestone-specific** requirements are defined in the corresponding `core/transitions/` file against the enablers that realize them. Solution files cite; neither restates the other.

## Transitions

Each transition file is the architect-owned **high-level architecture** of its delivery phase (goal, deltas keyed §N→§N, binding gates) — the milestone's vision slice and enabler package (→ SAFe correspondence). Deltas are enabler-shaped: type (`architectural` / `infrastructure` / `compliance` / `exploration`), a one-phrase statement, a benefit hypothesis, and acceptance criteria, sized to fit the milestone. Gap discipline (TOGAF): a § reviewed with no delta is recorded as carried-over, not omitted. Team-owned, permit-to-build-focused **low-level design** lives with the dev teams and is referenced, never stored here. Enablers are indexed canonically in `core/transitions/ENABLER-CATALOG.md` (name + description + *what & how to use*, plus the requirements each realizes), which links to each transition's *how-to-build* detail — the transition holds the build detail, the catalog the durable cross-transition view.

**Lifecycle.** One transition is open at a time, identified by its `Mn` prefix; close-before-create, carry-over, and promotion mechanics live in `core/transitions/README.md`. Creation and close are the architect's explicit calls (`core/AGENT-RULES.md` §9).
