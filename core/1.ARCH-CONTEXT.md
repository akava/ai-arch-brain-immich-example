## Problem statement (the top-level need)

The client (fictional, demo) operates self-hosted software for customers and wants to offer **managed Immich hosting** as a product: many isolated customer instances (later possibly shared infrastructure), operated to commercial availability expectations, at photo-library scales an individual self-hoster rarely reaches. Immich is built and documented primarily for single-node self-hosting; whether and how its architecture supports resilient, operable, multi-tenant-scale hosting is unknown to the client.

The engagement must first establish an evidence-backed picture of Immich's as-is architecture from its public documentation (components, integrations, data flows, operational characteristics), then design the scale-up: what a managed pilot deployment looks like, which quality attributes bound it (availability, ML-inference throughput, storage growth), which parts of the stock architecture carry over, and which need operator-side capabilities around them. The central tension: preserving Immich's stock upgrade path and single-codebase deployment model while meeting hosting-grade availability and scale the upstream docs do not target.
