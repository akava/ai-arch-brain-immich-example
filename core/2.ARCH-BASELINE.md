## 1. Reference Architecture & Patterns

### 1.1 Context & Scope

Evidence source: D1 discovery synthesis (`core/artifacts/logs/2026-07-11-immich-architecture-overview-synthesis.md`, T1–T2, T7).

- [Confirmed] Immich is a self-hosted photo and video management platform, documented and distributed primarily for single-node self-hosting; licensed AGPL v3 (`input/systems/immich/immich-repo-readme.md`).
- [Confirmed] The system uses a traditional client-server design: frontend clients communicate with backend services over HTTP using REST APIs; clients auto-generate their REST clients from the server's OpenAPI schema (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] Three first-party clients exist — mobile app (Android, iOS), responsive web app, and a CLI for bulk upload — all consuming the same OpenAPI-generated REST surface (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] Administrative functions (user management) are available only in the web app, not the mobile app (`input/systems/immich/immich-repo-readme.md`).
- [Confirmed] Durable state sits in three stores the server reaches through repository interfaces: Postgres (application data), Redis (job-queue state), and the file system (media); server, ML service, and web/CLI clients hold no durable server-side state, and the mobile app keeps a local on-device Isar database (`input/systems/immich/immich-architecture-overview.md`). Component roles are defined in §1.2.

### 1.2 Building Blocks

The as-built server-side containers (C4 container zoom). Canonical component names per `core/GLOSSARY.md`. Evidence source: synthesis T3–T7, conflict C1.

- [Confirmed] **immich-server** — TypeScript/Node.js service on the Nest.js framework (Express server, Kysely query builder), following hexagonal architecture that separates `src/repositories` (technology-specific) from `src/services` (core business logic). Handles REST API requests and executes background jobs (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **immich-machine-learning** — Python/FastAPI service to which all ML operations are externalized; can be deployed separately from the server or disabled entirely. Runs ONNX-format models, downloading/loading/caching them and reusing loaded models across requests, with a thread pool keeping inference off the async event loop; model settings are stored in Postgres. Models are large and memory-intensive (`input/systems/immich/immich-architecture-overview.md`; ADR-001).
- [Confirmed] **postgres** — persistent data store holding access/authorization data, users, albums, assets, sharing settings, and ML model settings (`input/systems/immich/immich-architecture-overview.md`). Stateful; the system of record.
- [Confirmed] **redis** — job-queue management via BullMQ; holds queue state (`input/systems/immich/immich-architecture-overview.md`). Stateful (operational queue state).
- [Confirmed] The `immich-server` container runs two workers — **api** (serves API/data/file requests for web and mobile) and **microservices** (handles most other work, e.g. thumbnail generation and video encoding, as *jobs*). Both run inside the single `immich-server` container by default (`input/systems/immich/immich-jobs-and-workers.md`).
- [Confirmed] Workers can be split across separate containers via environment variables: setting `IMMICH_WORKERS_INCLUDE: 'api'` on the original container and `IMMICH_WORKERS_EXCLUDE: 'api'` on a copied, port-stripped container renamed `immich-microservices` lets one container serve the API/UI while another runs background jobs (`input/systems/immich/immich-jobs-and-workers.md`; ADR-002).
- [Confirmed] Default packaging co-locates API serving and background-job processing in one `immich-server` container; a separate `immich-microservices` container is an optional split configuration, not the default (`input/systems/immich/immich-jobs-and-workers.md`; reconciles the two-container depiction in `input/systems/immich/immich-architecture-overview.md` — see synthesis C1).
- [Confirmed] `immich-machine-learning` can run on a **separate remote host**, reached over HTTP on port 3003; the server sends image previews to the remote container for processing (`input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] The remote ML container has **no built-in security measures**; the docs warn against running it on a public machine or paid cloud without added protection (`input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] Remote-ML offload is **partial**: Smart Search and Face Detection use the remote container, but Facial Recognition does **not** — it relies on model outputs stored in Postgres, making it a server-to-database operation that stays on the server side (`input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] Multiple remote-ML URLs are **processed sequentially with no load distribution**; a dedicated external load balancer is required to spread work across multiple ML containers (`input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] Server and remote-ML hosts must be **version-aligned** — mismatched versions may cause bugs and instability (`input/systems/immich/immich-remote-machine-learning.md`).

### 1.3 Integration Patterns

Evidence source: synthesis T1, T4, T6.

- [Confirmed] Client-to-server integration is synchronous HTTP/REST against a single OpenAPI-described surface; every client uses generated REST clients (`input/systems/immich/immich-architecture-overview.md`). Contract detail in §4.1.
- [Confirmed] Server-to-ML integration is synchronous HTTP: each request to the ML service carries the model task metadata and model name (`input/systems/immich/immich-architecture-overview.md`; ADR-001).
- [Confirmed] Background work is queue-based: the server enqueues jobs to Redis via BullMQ, and the microservices worker consumes them. Some jobs trigger subsequent jobs — Smart Search and Facial Recognition automatically execute after Thumbnail Generation completes (`input/systems/immich/immich-architecture-overview.md`, `input/systems/immich/immich-jobs-and-workers.md`; ADR-002).

### 1.4 Data Architecture & Flows

Evidence source: synthesis T6, T7.

- [Confirmed] Uploading a new asset kicks off a chain of jobs — metadata extraction, thumbnail generation, machine-learning tasks, and storage-template migration (if enabled) (`input/systems/immich/immich-jobs-and-workers.md`). The documented job set also includes video transcoding, smart search, facial recognition, sidecar (XMP), and deletion jobs (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] Job ordering is partly dependency-driven: thumbnail generation runs first for a newly uploaded file, and Smart Search and Facial Recognition run after it completes (`input/systems/immich/immich-architecture-overview.md`, `input/systems/immich/immich-jobs-and-workers.md`).
- [Confirmed] Some jobs run on a schedule (default nightly at midnight), configured via System Settings → Nightly Tasks; External Libraries scanning and Database Dump jobs are configured separately (`input/systems/immich/immich-jobs-and-workers.md`).
- [Confirmed] Media assets are stored on the file system and reached through repository interfaces; application data (users, albums, assets, sharing, authorization, ML model settings) persists in Postgres (`input/systems/immich/immich-architecture-overview.md`). Store roles in §1.2.
- [Confirmed] On-disk media layout is governed by a configurable **storage template** — a Handlebars pattern of directory/filename variables (e.g. `{{y}}`, `{{MM}}`, `{{album}}`, `{{filename}}`) with conditionals; the default is `Year/Year-Month-Day/Filename.Extension` (`input/systems/immich/immich-storage-template.md`).
- [Confirmed] Storage-template date/time variables render in the **server's local timezone**; the system appends sequence numbers to prevent filename collisions, and when an asset is in several albums the most recently created album name fills `{{album}}` (`input/systems/immich/immich-storage-template.md`).
- [Confirmed] Changing the storage template is not retroactive by itself: the **Storage Template Migration** job applies the new pattern to the existing library (`input/systems/immich/immich-storage-template.md`).
- [Confirmed] Postgres schema changes are delivered as **TypeScript migrations** (generated via `mise` from `server/src/schema`); any new migrations are **applied automatically at server startup**, and a single-step revert rolls back the latest migration (`input/systems/immich/immich-database-migrations.md`).

## 2. Tech Stack & Platforms

The products documented as running today. Versions are not stated in the D1 doc set (an observed gap, → §7). Evidence source: synthesis T2, T3, T4. Component names per `core/GLOSSARY.md`.

- [Confirmed] **Server runtime:** TypeScript on Node.js, Nest.js framework, Express server, Kysely query builder; deployed as Docker containers (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Machine-learning runtime:** Python with FastAPI; models in ONNX format (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Data store:** Postgres (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Queue / cache:** Redis, used via BullMQ for job queues (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Mobile client:** Dart/Flutter, with Isar local database and Riverpod state management (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Web client:** TypeScript on SvelteKit with Tailwind CSS (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **CLI:** an npm package that drives the instance via the API, chiefly for bulk asset upload (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **Packaging:** components run as individual Docker containers (`input/systems/immich/immich-architecture-overview.md`).
- [Confirmed] **ML hardware acceleration (experimental):** Smart Search and Facial Recognition can be GPU-accelerated via one of five backends — ARM NN (Mali), CUDA (NVIDIA), ROCm (AMD), OpenVINO (Intel), or RKNN (Rockchip) — enabled through the `hwaccel.ml.yml` compose file; supported only on Linux and Windows-via-WSL2 servers (`input/systems/immich/immich-ml-hardware-acceleration.md`).
- [Confirmed] **Multi-GPU ML:** work spreads across GPUs by setting `MACHINE_LEARNING_DEVICE_IDS` to a comma-separated device list and `MACHINE_LEARNING_WORKERS` to the device count (`input/systems/immich/immich-ml-hardware-acceleration.md`).
- [Confirmed] **Hardware video transcoding (experimental):** GPU transcoding is available via NVENC (NVIDIA), Quick Sync (Intel), RKMPP (Rockchip), or VAAPI (AMD/NVIDIA/Intel), enabled through the `hwaccel.transcoding.yml` compose file; supported only on Linux and Windows-via-WSL2 (WSL2 does not support Quick Sync; Raspberry Pi unsupported) (`input/systems/immich/immich-hardware-transcoding.md`).

### 3.1 Performance & Throughput

Anchored to **R-03**, **OQ#2**; pending candidate NFR-003/NFR-004/NFR-005 (QA-3/QA-5/QA-6). Observed values (capacity *floors* and the ML memory profile are in §3.4 — referenced, not duplicated):

- [Confirmed] ML scale-out across multiple registered remote URLs is **processed sequentially with no load distribution**; an external load balancer is required to spread work — scale-out is an operator-supplied capability, not a stock feature (§1.2; `input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] Hardware transcoding is **encoding-only by default** — the CPU still performs decoding and tone-mapping, so GPU transcoding does not eliminate CPU load — and typically produces **larger, lower-quality** output than software transcoding (a storage-growth and quality effect) (§3.4; `input/systems/immich/immich-hardware-transcoding.md`). Per-node transcode throughput and the storage-growth factor remain unquantified.
- [Confirmed] Per-node/per-GPU ML **throughput, concurrency limits, and per-model memory footprint remain unquantified** — only floors exist (§3.4, OQ#2, R-03).
- [Confirmed] Documented ML-job concurrency guidance: **2–3 concurrent jobs can max out a normal (CPU) machine**; with a GPU concurrency can be raised more comfortably **but should still not exceed 16 in most cases** (`input/systems/immich/immich-faq-scaling-excerpts.md:67`). To reduce facial-recognition load, the model can be switched to the lighter **`buffalo_s`** in place of the default `buffalo_l`, and transcoding threads set to **1–2** (`input/systems/immich/immich-faq-scaling-excerpts.md:49`). These are the only first-party throughput *numbers* on record — still coarse operator heuristics, not per-node capacity figures (OQ#2 stays open).

### 3.3 Availability & Resilience

Anchored to **R-01**, **R-05**, **OQ#1**; pending candidate NFR-001/NFR-002 (QA-1/QA-2). Observed values:

- [Confirmed] The three stateful stores (Postgres system-of-record, Redis queue state, filesystem media — §1.1/§1.2) have **no documented HA, replication, or failover path** in the first-party doc set; availability risk concentrates entirely in these stores (server/ML/web/CLI hold no durable server-side state) (`input/systems/immich/immich-architecture-overview.md`; synthesis T5/C1). A documented *silence*, not an asserted control.
- [Confirmed] Postgres schema migrations are **applied automatically at server startup**, coupling schema evolution — and thus the upgrade window — to the server-container lifecycle; a single-step revert rolls back only the latest migration (§1.4; `input/systems/immich/immich-database-migrations.md`). No rolling / zero-downtime migration path against a shared Postgres is documented. Startup migrations can be skipped via `DB_SKIP_MIGRATIONS` (default `false`) (`input/systems/immich/immich-environment-variables.md:60`) — an operator lever, not a documented rolling-upgrade path.
- [Confirmed] Remote-ML hosts must be **version-aligned** with the server, coupling ML-tier upgrades to server upgrades (§1.2; `input/systems/immich/immich-remote-machine-learning.md`).
- [Confirmed] **Backup posture (partial, DB-only automation).** Immich recommends a **3-2-1** strategy and automatically dumps the database **daily at 2:00 AM**, keeping the **last 14 backups** in `UPLOAD_LOCATION/backups`; these dumps are **metadata only — no photos or videos** (`input/systems/immich/immich-backup-and-restore.md:9,15,17`). **Filesystem/media backup is not managed by Immich** — it is operator-supplied over `UPLOAD_LOCATION/library|upload|profile` (`input/systems/immich/immich-backup-and-restore.md:42,44-46`). For consistency the docs require **stopping the `immich-server` container, or dumping the database first and the filesystem second**, so the DB never references assets missing from the filesystem backup (`input/systems/immich/immich-backup-and-restore.md:62`). Database checksums are on by default for new installs **since v1.104.0**, with `pg_amcheck` available for integrity scans (`input/systems/immich/immich-faq-scaling-excerpts.md:77,94`). No documented backup of Redis queue state.

### 3.4 Scalability & Capacity

Hardware **floors** for ML acceleration are documented, but per-node throughput and concurrency remain unquantified (→ OQ#2, R-03). Observed floors:

- [Confirmed] **CUDA** ML acceleration requires a GPU of **compute capability 5.2 or higher** and **NVIDIA driver ≥ 545** (CUDA 12.3 support) (`input/systems/immich/immich-ml-hardware-acceleration.md`).
- [Confirmed] **ROCm** ML acceleration requires **≥ 35 GiB free disk space**, and — because MIGraphX compiles models at runtime — the initial inferences are slow (`input/systems/immich/immich-ml-hardware-acceleration.md`).
- [Confirmed] **RKNN** (RK3576/RK3588): `MACHINE_LEARNING_RKNN_THREADS` set to **2–3** is the recommended value for improving performance (`input/systems/immich/immich-ml-hardware-acceleration.md`).
- [Confirmed] Hardware **transcoding** is **encoding-only by default** (CPU still handles decoding and tone-mapping) and typically produces **larger, lower-quality** output than software transcoding; two-pass mode works only with NVENC, and NVIDIA/AMD GPUs cannot encode VP9 (`input/systems/immich/immich-hardware-transcoding.md`).
- [Confirmed] **ML tuning knobs (defaults).** `MACHINE_LEARNING_WORKERS` defaults to **1**; `MACHINE_LEARNING_MODEL_TTL` (model unload idle timeout) defaults to **300 s**; `MACHINE_LEARNING_REQUEST_THREADS` defaults to the **CPU core count**; intra/inter-op threads default **2/1**; OCR max batch size **6**; `MACHINE_LEARNING_DEVICE_IDS` defaults to **`0`** (`input/systems/immich/immich-environment-variables.md:83,86-89,101,103`). These are the documented capacity levers for the ML tier; none specifies a resulting throughput (OQ#2).
- [Confirmed] **Horizontal scaling requires shared stores.** Parallel `immich-server` instances must all connect to the **same Postgres and Redis** and have the **same files mounted** into every container — all state lives in Postgres, Redis, and the filesystem (`input/systems/immich/immich-scaling-guide.md:9,23`). On a **single machine, running multiple containers is unlikely to give any benefit** (`immich-scaling-guide.md:13`); when scaling only for background work, the **API worker can be disabled** on the extra instances (`immich-scaling-guide.md:19`), and on Kubernetes scaling "can be as easy as incrementing the amount of replicas" (`immich-scaling-guide.md:15`). **No capacity guidance:** the scaling guide states it gives **no specific instructions** and leaves the details "as an exercise for the reader" (`immich-scaling-guide.md:15`) — a documented silence on sizing, not a sizing method.
- [Confirmed] **Supported / unsupported storage.** External libraries can be mounted **read-only (`:ro`)** and over **CIFS/Samba**, but **NTFS and FAT/exFAT/FAT32 filesystems are not supported** (`input/systems/immich/immich-faq-scaling-excerpts.md:29,33,37,73`) — a hard constraint on the shared-filesystem tier a 50×5 TB estate depends on. Duplicate checking exists only for upload libraries (by file hash) and is **per-library, not global** (`immich-faq-scaling-excerpts.md:25`).

## 6. Deployment & Environments

Evidence source: D3 synthesis (`core/artifacts/logs/2026-07-11-immich-scale-up-design-synthesis.md`, T14–T15).

- [Confirmed] **Docker Compose is the recommended production deployment method**, installed in three steps (download the compose + `.env` files, populate `.env`, `docker compose up`) (`input/systems/immich/immich-docker-compose.md:9`). Required `.env` keys include `UPLOAD_LOCATION` (asset directory with sufficient free space), `DB_PASSWORD` (**constrained to `A-Za-z0-9` only**, no special characters), `TZ`, and `IMMICH_VERSION` (defaults to `v3`, pinnable to a specific release e.g. `v2.1.0`) (`immich-docker-compose.md:33-36`). Docker Engine **earlier than v25** requires commenting out the DB `start_interval` line (`immich-docker-compose.md:51`).
- [Confirmed] **Kubernetes is supported only via the official `immich-charts` Helm chart** (community examples at kubesearch.dev); the doc carries **one operational caveat and no sizing guidance** — the Alpine-based container images hit a **DNS-resolution bug on clusters whose host nodes set a search domain** (`resolv.conf`) (`input/systems/immich/immich-kubernetes.md:9,11,15`). The k8s support surface is thin: a chart pointer plus one caveat, no capacity or HA guidance.
- [Confirmed] **DB/Redis config is required on every worker.** All `DB_` variables and all `REDIS_` variables must be provided to **all** workers, including both `api` and `microservices` (`input/systems/immich/immich-environment-variables.md:63,77`) — a hard coupling for any split-worker or multi-instance topology. Worker selection is via `IMMICH_WORKERS_INCLUDE` / `IMMICH_WORKERS_EXCLUDE` (§1.2); the vector extension is selectable via `DB_VECTOR_EXTENSION` (`vectorchord` or `pgvector`) (`immich-environment-variables.md:59`). Destroying an instance's volumes and data is `docker compose down -v` (`input/systems/immich/immich-faq-scaling-excerpts.md:107`).
