# 3.ARCH-TARGET.md — Target Architecture

## Purpose

`3.ARCH-TARGET.md` is the source of truth for the **target architecture** — the system rules this brain validates work against: reference architecture, sanctioned technologies, approved patterns, NFR baselines, and conventions. It defines what "good" architecture looks like here.

No structured output may contradict this file. Agents check here first (reuse before create).

The as-is counterpart is `core/2.ARCH-BASELINE.md`; phase movement lives in `core/transitions/`.

**Altitude:** scoped to system, data, and integration architecture — plus delivery and operating constraints that bind the system design (§5). Detail owned by other brains is referenced, not duplicated (see [Related brains](#related-brains)).

- [Tentative] **Customer-library onboarding — ingest data flow.** Customer libraries enter a managed instance through one of Immich's documented import paths (BASELINE §1.4); the target rule is that onboarding **defaults to CLI-upload ingest** — assets copied into `UPLOAD_LOCATION`, **quota-counted** (NFR-005), governed by the **single storage template** class, and subject to always-on server-side deduplication (idempotent re-run in lieu of a resume flag). The pilot default is **staged operator-driven CLI ingest**: customer data lands on an operator staging volume, the operator runs the upload into the instance, then staging is retired — decoupling the bulk transfer from the front-door reverse proxy (`50000M`/`600s`) and keeping the `--concurrency` **arrival-rate lever operator-side** (the one control over the NFR-003 ML-backlog arrival profile — both onboarding modes feed the same scan-triggered thumbnail→ML job chain, so mode does not change the backlog physics, only the arrival rate). **Direct CLI upload** is allowed for small libraries. **External-library-based ingest is not the default** — it is quota-exempt (defeats NFR-005 accounting — R-09), single-user-unchangeable-owned, and carries the path-removal-deletes hazard; it is an opt-in exception only, and when taken adds a **second, in-place storage class** outside the storage template that the §3.4 backup design (E-03) must cover. The arrival-format ingest-and-map procedure is **operator procedure**, undocumented upstream (OQ#6) — formats are not specified here. Realization direction: an operator **customer onboarding pipeline** capability (E-06). See §1.5 (rollout rule) and §3.4 (storage-class/backup implication). (ADR-008, proposed.)

### 3.1 Performance & Throughput

- [Tentative] **Adopts NFR-003.** The ML tier requires a capacity design that can sustain ≈ **3.86 assets/sec aggregate** (the 1M-in-72h target) — but the throughput figure is **[TBD]**: no per-asset inference time exists in the doc set (OQ#2, R-03), so the design must be **benchmarked before commit**. Derived acceptance bound (nfr-analysis §C): mean ≤ ~4.15 s/asset at the FAQ's ≤16 GPU-concurrency ceiling on a single node, or proportional horizontal ML scale-out. Response direction: **requires GPU-class ML capacity and a benchmarking gate**, sized against measured per-asset time — not a fixed node count set on assumptions. Realization direction: a **dedicated remote-ML tier** sized by the benchmark gate — tier size and the 72 h commitment stay [TBD] until per-asset inference time is measured (ADR-004, proposed).
- [Tentative] **Adopts NFR-004.** Stock multi-URL ML dispatch is **sequential with no load balancing** (BASELINE §3.1). Response direction: the scale-out topology **requires an operator-supplied load-balancing capability** with health-checking and retry across ML nodes to reach the NFR-003 aggregate; account for the **server-bound Facial Recognition** portion that cannot leave the server (BASELINE §1.2). Balances against the added LB failure surface (ties §3.3). (QA-5, R-03, OQ#1.) Realization direction: an **operator-run, health-checked load balancer fronting the dedicated ML tier**, reached by the server as a single ML URL (ADR-004, proposed).

### 3.2 Observability & Operability

- [Tentative] **Fleet observability & health capability.** Immich ships observability *primitives without a control plane* (BASELINE §3.2): opt-in Prometheus/OTel metrics on 8081/8082 (groups `api`/`host`/`io`, off by default), env-var-only JSON logging with no routing/retention/alerting, and a one-shot `.immich` startup marker-file check as the only first-party health mechanism — no container-healthcheck guidance (OQ#5). Response direction: to hold **NFR-001** across 50 instances, the estate **requires an operator-run central fleet observability plane** — fleet-wide telemetry (a baseline instance-config setting), Prometheus-compatible scrape of every instance's 8081/8082, central shipping + defined **retention** of the JSON log stream, operator-built **synthetic health probes + container healthchecks** over the stock startup gate, and **alerting tied to NFR-001's error budget** — with per-instance data **segregated** per the isolated-instance philosophy (ADR-003 alignment). The front-door proxy's `600s` timeout / `50000M` upload cap (BASELINE §3.2, T26) are a **monitored surface + alerting-relevant thresholds**, not re-decided here. End-to-end tracing is **out of scope** for the pilot — the docs' OTel mention is metrics-export only, no documented per-request span export. OQ#5 is **partially resolved**: log routing/retention decided; **container-health/probe design stays [TBD]** as M1 build work, and the detection window stays **[TBD]** until set at the pilot gate. Realization direction: an operator monitoring/alerting stack + log pipeline + health probes wrapping the stock instances (ADR-007, proposed).

### 3.3 Availability & Resilience

- [Tentative] **Adopts NFR-001.** Stock provides no HA for the three shared stateful stores (BASELINE §3.3; R-01 critical, OQ#1), and the 99.9% budget is **≤ 43.8 min/month**. Response direction: the topology **requires operator-supplied high-availability for all three stores** (Postgres, Redis, filesystem media) — the availability target has no path on stock alone. HA design + sizing need evidence beyond the doc set (OQ#1). Bought against operability and CON-001 (each added tier erodes the stock path). Realization direction: **operator-managed per-instance HA Postgres/Redis** (endpoint-transparent to the stock containers; shared cross-instance store pools rejected for blast radius — T14 SPOF), with a **failover drill as a pilot gate**; the filesystem-media store's protection realizes via the §3.4 backup posture (ADR-003, proposed; media: ADR-005, proposed).
- [Tentative] **Adopts NFR-002.** Migrations apply at server startup and the ML tier is version-aligned (BASELINE §3.3), with no stock rolling/zero-downtime path. Response direction: the upgrade procedure **requires a controlled maintenance-window capability** covering server + version-aligned ML restart, with migration duration **measured at pilot scale** before the ≤15 min window is confirmed (the fits-window sub-claim is unverifiable from evidence — R-05, QA-2). The ≤15 min planned downtime also **spends ~34% of the NFR-001 monthly budget** per upgrade — the two targets are co-constrained.
- [Tentative] **Adopts CON-001.** The stock upgrade-path / single-codebase model is the reversibility bound on §3.1/§3.3 responses: operator-added HA, LB, and upgrade machinery **must stay minimal and reversible** so the stock upgrade path and version-alignment are preserved (R-05, QA-2/QA-8). This constraint governs how far the above capabilities may diverge from stock; §1.5 and §5 carry its coexistence/gating realization when populated.

### 3.4 Scalability & Capacity

- [Tentative] **Adopts NFR-005.** Aggregate media capacity is **≥ 250 TB** (5 TB × 50), on a filesystem tier where **NTFS/FAT are unsupported** (BASELINE §3.4) and **media backup is not automated by Immich** (DB-only-automated, metadata only — BASELINE §3.3, T18). Response direction: the storage design **requires (a) an operator-supplied media-backup capability** honoring the fixed DB-first-then-filesystem ordering, **(b) Redis-queue-state durability** (stock backs up neither), and **(c)** provisioning headroom above 250 TB for the **unquantified GPU-transcode storage-growth multiplier** (BASELINE §3.1/§3.4). Per-instance sizing has no first-party guidance — sizing is an operator capability, not a stock method. Realization direction: **per-instance dedicated media volumes on supported (non-NTFS/FAT) filesystems + operator snapshot/replication backup** honoring the DB-first-then-filesystem ordering; headroom % stays [TBD] until transcode-growth telemetry exists; Redis durability realizes via the §3.3 store-HA posture (ADR-005, proposed; Redis: ADR-003, proposed).

### 3.5 Security & Compliance

- [Tentative] **Adopts NFR-006.** The remote-ML channel carries user photo previews over port 3003 with **no built-in security** (BASELINE §1.2, OQ#3). Response direction: the ML channel **requires authentication, transport encryption, and network isolation** capabilities (operator-supplied) to a defined baseline, delivered as a **pilot gate** for transition M1. Adds latency/surface to the synchronous ingest path (ties §3.1) and may constrain how freely the ML tier scales behind the required LB (QA-7, OQ#3). Realization direction: **auth + TLS terminated at a reverse proxy in front of port 3003**, combined with the §3.1 ML-tier load balancer on an isolated network segment (ADR-004, proposed).
- [Tentative] **Identity & access — per-customer federation.** Immich's OIDC surface is first-party and per-instance (BASELINE §1.6, T19); the managed estate needs a per-customer identity posture, not a shared one. Response direction: each customer instance **requires federation to that customer's own IdP** via the documented per-instance OIDC config (issuer/client/scope + `immich_role`/`immich_quota`/`preferred_username` claim mappings), **with local password login retained** as a break-glass/admin fallback rather than OIDC-only. An operator-central IdP is rejected as a cross-tenant identity SPOF (same blast-radius argument as ADR-003; ties §1.6). Because claims are applied **at creation only, never re-synced** (BASELINE §1.6), the design **requires an operator role/quota/label reconciliation procedure** (R-07); the retained password path **requires a lockout/break-glass runbook** via the `immich-admin` CLI (R-08). Caveats carried, not resolved here: the `immich_quota` claim is an upload gate that **exempts external libraries** (R-09, ties §3.4 capacity accounting); and there is **no documented API-key surface or per-session forced-logout** — [TBD] under **OQ#4**. Realization direction: **per-instance OIDC to the customer IdP + retained password fallback**, delivered with the R-07 reconciliation and R-08 break-glass runbooks (ADR-006, proposed).
